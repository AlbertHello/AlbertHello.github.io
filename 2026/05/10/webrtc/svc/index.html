SFU 服务端做 SVC 丢层适配：从“能转发”到“能抗弱网”

本文最后更新于 2026-05-20T21:10:39+08:00

在 SFU 服务端做 H.264 SVC 丢层适配：从“能转发”到“能抗弱网”

在实时音视频系统里，SFU 的优势很明确: 服务端不解码、不合流、不重新编码，只负责把上行 RTP 流选择性转发给不同的下行用户。这种架构延迟低、成本可控，也更适合直播和互动场景。

但它也带来一个典型问题: 当某一个拉流端网络变差时，服务端不能简单地把上游编码器码率调低。因为同一条上行流通常会被多个拉流端共享，如果为了一个弱网用户降低上游编码质量，其他网络正常的用户也会被一起降级。

NMRTC 在这类场景里采用的思路是: 让推流端输出 H.264 SVC 分层码流，服务端在 RTP 层识别基础层和增强层。网络正常时转发完整码流；某一路下行变差时，只对这一条下行链路丢弃部分增强层，从而降低带宽压力，同时不影响其他拉流端。

这篇文章不展开所有实现细节，只讨论服务端做 SVC 丢层适配时最关键的几个工程问题。

为什么 SFU 需要 SVC

SVC 是 Scalable Video Coding，可伸缩视频编码。它把一条视频码流拆成多层:

  • 基础层: 保证最低可解码的视频内容。
  • 增强层: 在基础层之上提升清晰度、帧率或质量。

对 SFU 来说，SVC 的价值不只是“编码更高级”，而是让同一条上行码流可以适配不同下行网络:

服务端仍然不需要解码，也不需要为每个用户重新编码，只是在 RTP 转发阶段做分层选择。这样既保留了 SFU 的低延迟和低成本，也给弱网适配留下了空间。

服务端能识别什么

H.264 SVC 的 RTP 包里会携带层级信息。NMRTC 服务端主要关注两个字段:

字段 含义 适配价值
DID 空域依赖层 用来区分基础空间层和空间增强层
TID 时域层 用来区分基础帧率层和时间增强层

可以粗略理解为:

  • DID=0, TID=0: 最基础的视频层，弱网下也尽量保留。
  • TID>0: 提升帧率的时间增强层，弱网时可以优先丢。
  • DID>0: 提升清晰度或码率的空间增强层，带宽紧张时可以丢。

在实际 RTP 封包里，服务端会遇到 Single NAL、STAP-A、FU-A 等不同形态。难点不在于读出几个字段，而在于不同封包方式下层级信息的位置并不总是一样。例如 Prefix NAL 可能单独携带层级信息，而真正的视频 slice 需要继承同一帧里的 Prefix 信息；FU-A 分片则必须保证一个 NAL 的所有分片要么都保留，要么都丢弃。

因此，服务端不能只按单个 RTP 包即时决策，而需要按 frame 做一次相对完整的判断。

丢层不是简单丢包

很多人第一次做 SVC 适配时，容易把问题理解成“识别增强层，然后丢掉它”。这只完成了一半。

RTP 接收端依赖 sequence number 判断丢包、乱序和重传。如果服务端直接丢掉增强层包，下行接收端看到的就是 seq 空洞:

拉流端无法知道这是服务端主动做的策略丢层，只会认为网络丢包了。结果通常是 jitter buffer 等待缺失包，NACK 模块持续请求重传，甚至把弱网链路进一步打满。

所以，真正可用的服务端 SVC 丢层至少要同时处理三件事:

  1. 只丢弃可被丢弃的增强层，不破坏基础层和 NAL 分片完整性。
  2. 对保留下来的 RTP 包重写连续的下行 seq，避免接收端误判丢包。
  3. 当真实网络丢包发生时，能把下行 NACK 翻译回上游原始 seq，并正确恢复 RTX。

这也是 SVC 适配从“理论可行”走向“线上可用”的关键分界线。

其中 UplinkStream 负责接收和缓存上行 RTP，NmrtcStreamManager 负责把同一路流分发给多个 DownlinkStream。每个 DownlinkStream 代表一路下行连接，也就是一个具体的拉流端。

SVC 丢层策略放在 DownlinkStream 侧，而不是 UplinkStream 侧。原因很直接: 弱网状态是下行链路独有的。用户 A 网络正常时可以看完整码流，用户 B 网络较差时可以只看基础层，两者不能互相影响。

因此，每个 DownlinkStream 都维护自己的 SVC 状态:

  • 当前丢层策略。
  • 本路下行的 seq 重写映射。
  • 已经被策略丢弃的上游 seq 集合。
  • NACK 翻译和 RTX 恢复所需的上下文。

这让 SVC 策略天然具备 per-user 粒度。后续如果把下行拥塞控制、pacer 或 BWE 结果接入进来，就可以由每一路 DownlinkStream 根据自己的网络状态选择不同的层级。

几种实用的降级策略

在工程实现上，没必要一开始就追求复杂的自适应算法。更可靠的做法是先把几个稳定的策略跑通:

策略 保留内容 适合场景
keep_all 所有层 网络正常
drop_temporal 保留基础时间层 带宽轻微紧张，希望先降帧率
drop_spatial 保留基础空间层 带宽明显紧张，希望降低码率
keep_base_only 只保留基础层 严重弱网，需要优先保证可播放

这里的取舍不是绝对的。丢时间层通常会牺牲流畅度，但画面清晰度变化相对小；丢空间层通常能更明显降低码率，但可能影响清晰度。实际系统里应该结合编码器输出结构、业务场景和接收端表现来调策略。

对于 NMRTC 当前的直播场景，比较合理的演进路线是:

  1. 正常网络下保持完整码流。
  2. 轻微拥塞时优先丢时间增强层，降低帧率压力。
  3. 持续拥塞时丢空间增强层或只保留基础层。
  4. 网络恢复后等待关键帧，再逐步恢复更高层。

seq 重写和 NACK 翻译是核心

这要求服务端在发送每个保留包时记录双向映射:

  • source seq -> downlink seq: 用于后续 RTX 重传恢复。
  • downlink seq -> source seq: 用于把接收端 NACK 翻译回上游缓存。

当拉流端真的在网络中丢了一个包，它 NACK 的是自己看到的 downlink seq。服务端需要先把它翻译成 source seq，再从 UplinkStream 的上游 RTP 缓存里找原始包。

同时，服务端还要识别“这个包本来就是策略丢掉的增强层”。这类包不能因为 NACK 又被重传出去，否则丢层策略会被破坏，还会把弱网链路重新压垮。

也就是说，一套完整的 SVC 丢层链路必须同时区分两类“缺包”:

  • 服务端主动策略丢弃: 不应该重传。
  • 下行网络真实丢包: 应该翻译 seq 并走 RTX 恢复。

这部分处理不好，表面上看是 SVC 丢层，实际播放体验可能会更差。

关键帧和层级恢复

从低层级恢复到高层级时，还会遇到解码参考关系的问题。尤其是空间层切换后，接收端可能短时间内缺少完整参考帧，导致花屏、等待或恢复慢。

NMRTC 在强降级或恢复强降级策略时，会向上游请求关键帧。这样做不是 SVC 丢层的必要条件，但能显著提升策略切换后的稳定性。对实时系统来说，切换策略能生效只是第一步，切换后尽快回到可观看状态才是最终目标。

小结

SVC 给 SFU 带来的核心能力，是让服务端在不解码、不重新编码的前提下，对不同下行用户做差异化带宽适配。但服务端 SVC 丢层并不是简单识别字段然后丢包，它涉及 RTP frame 级处理、NAL 分片完整性、marker 修正、seq 重写、NACK 翻译和 RTX 恢复。

从 NMRTC 的实践看，比较稳妥的实现路径是先把固定策略做正确，再接入下行拥塞控制和 pacer 状态，让每一路 DownlinkStream 根据自己的网络质量独立选择 SVC 层级。这样既符合 SFU 架构的低成本转发模型，也能为后续更完整的弱网自适应打好基础。
