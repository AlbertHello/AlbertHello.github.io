webrtc之RtpTransceiver详解

本文最后更新于 2026-06-23T17:28:57+08:00

阅读本文之前建议先读： webrtc 之创建 peer_connection、 WebRTC 之 SetLocalDescription

阅读本文之后你将了解：

  1. RtpTransceiver 是什么？

  2. RtpTransceiver 和 RtpSender、RtpReceiver、VoiceChannel、VideoChannel 是什么关系？

  3. SetLocalDescription 时为什么会创建 VoiceChannel / VideoChannel？

  4. media engine 创建的 send/receive channel 和 WebRTC pc 层的 Channel 是什么关系？

RtpTransceiver 是什么？

RtpTransceiver 是 Unified Plan 中非常关键的对象。它不是网络收发类，也不是 SDP 文本对象，而是把一个 m 行对应的 RTP 发送、接收、媒体通道和底层传输通道串起来的对象。

官方源码注释说得很直接：

也就是说：

  • audio 类型的 RtpTransceiver 持有 audio sender、audio receiver，最终绑定一个 VoiceChannel。

  • video 类型的 RtpTransceiver 持有 video sender、video receiver，最终绑定一个 VideoChannel。

  • 一个 transceiver 通常对应 SDP 中的一个 m 行，mid_ 就是该 m 行的 mid。

重要成员变量

  • media_type_：区分 audio / video。它决定后面创建 VoiceChannel 还是 VideoChannel。

  • mid_：和 SDP m 行绑定的 mid。UpdateTransceiverChannel 传入的 content.mid() 最终会保存到这里。

  • senders_：当前 transceiver 管理的 RtpSender 列表。Unified Plan 下通常是一个 sender。

  • receivers_：当前 transceiver 管理的 RtpReceiver 列表。Unified Plan 下通常是一个 receiver。

  • channel_：真正和 RTP media channel、RTP transport 关联的 pc 层通道对象。audio 是 VoiceChannel，video 是 VideoChannel。

  • transport_name_：SetChannel 绑定 RtpTransportInternal 后保存的 transport name，一般就是 mid。

  • owned_send_channel_ / owned_receive_channel_：有些构造路径会提前创建 media engine 的 send/receive channel，等 CreateChannel 时再包装成 VoiceChannel 或 VideoChannel。

  • media_engine_ref_：用于在 worker thread 上安全访问 ConnectionContext 里的 media engine。

  • context_：拿三大线程、media engine、ssrc generator 等公共资源。

RtpTransceiver 的创建时机

RtpTransceiver 的创建来源主要有两类：

  • 应用层主动添加媒体：AddTrack 或 AddTransceiver 会创建 sender、receiver 和 transceiver。

  • SDP 协商过程中根据 m 行补齐：SdpOfferAnswerHandler::UpdateTransceiversAndDataChannels 会把 m 行和已有或新建的 transceiver 关联起来。

在 PeerConnection 构造函数里还会针对 Plan B 创建默认 audio/video transceiver，但 M146 主线推荐 Unified Plan，后面重点按 Unified Plan 理解即可。

SetLocalDescription 中如何走到 RtpTransceiver？

关键调用链：

  • PeerConnection::SetLocalDescription

    • SdpOfferAnswerHandler::SetLocalDescription

      • SdpOfferAnswerHandler::ApplyLocalDescription

        • SdpOfferAnswerHandler::UpdateTransceiversAndDataChannels

          • SdpOfferAnswerHandler::AssociateTransceiver

          • SdpOfferAnswerHandler::UpdateTransceiverChannel

            • RtpTransceiver::CreateChannel

UpdateTransceiverChannel 本身不直接创建 VoiceChannel/VideoChannel，它只是判断当前 content 是否 rejected，以及这个 transceiver 是否已经有 channel。

所以这一步准确地说是：SetLocalDescription 应用 SDP 时，为每个未拒绝的 audio/video m 行确保 RtpTransceiver 拥有一个对应的 VoiceChannel 或 VideoChannel。

CreateChannel 具体做了什么？

RtpTransceiver::CreateChannel 分两段：

  • 第一段切到 worker thread。如果构造 transceiver 时已经提前创建了 owned_send_channel_ / owned_receive_channel_，这里会优先复用；否则通过 media engine 创建新的 send/receive channel，并包装成 VoiceChannel 或 VideoChannel。

  • 第二段调用 SetChannel，切到 network thread，把 channel 绑定到前面 JsepTransportController 创建好的 RtpTransportInternal。

核心代码：

CreateMediaContentChannels

CreateMediaContentChannels 创建的是 media engine 层的收发通道，不是 pc 层的 VoiceChannel/VideoChannel。

这里可以这样理解：

  • media engine 的 send channel 负责更靠近编码、发送 RTP 参数、发送流控制的逻辑。

  • media engine 的 receive channel 负责更靠近解码、接收 RTP、接收流控制的逻辑。

  • VoiceChannel/VideoChannel 是 pc 层的包装对象，负责把 media channel 和 RtpTransportInternal 连接起来。

CreateMediaChannel

CreateMediaChannel 是一个模板函数，用来把 media engine 层的 send/receive channel 包装成 pc 层 channel。

对应关系：

  • audio：CreateMediaChannel<VoiceChannel, VoiceMediaSendChannelInterface, VoiceMediaReceiveChannelInterface>

  • video：CreateMediaChannel<VideoChannel, VideoMediaSendChannelInterface, VideoMediaReceiveChannelInterface>

所以，不能把 media engine 创建的 send/receive channel 和 pc 层的 VoiceChannel/VideoChannel 混为一个对象。它们是上下两层。

SetChannel 绑定 RTP transport

CreateChannel 创建出 VoiceChannel/VideoChannel 之后，还不能直接收发 RTP。它必须绑定到 RtpTransportInternal。

这个 transport 是前面 SetLocalDescription 的 PushdownTransportDescription 阶段通过 JsepTransportController::MaybeCreateJsepTransport 创建出来的。

SetChannel 的关键逻辑：

重点：

  • SetChannel 在 signaling thread 发起。

  • transport_lookup 在 network thread 上执行。

  • channel->SetRtpTransport(transport) 之后，RTP/RTCP 包才有传输通道。

  • 设置 first packet received/sent、packet received callback 也是在这里完成的。

PushNewMediaChannel

RtpTransceiver 持有 sender/receiver，但 sender/receiver 真正工作时需要知道 media channel。

PushNewMediaChannel 的作用就是把 channel 里的 send/receive media channel 推给所有 sender/receiver。注意：M146 的 CreateChannel() 调用 SetChannel(..., /*set_media_channels=*/false)，主线创建 channel 时通常不会再走 PushNewMediaChannel()；因为 media channel 要么在 transceiver 构造阶段已经传给 sender/receiver，要么在 CreateChannel() 的 worker thread 段已经通过 SetMediaChannels() 设置过。

这也是为什么 RtpTransceiver 是一个“装配层”：

  • 它不自己编码音视频。

  • 它不自己收发 UDP 包。

  • 它把 sender/receiver、media engine channel、VoiceChannel/VideoChannel、RtpTransport 这些对象按 SDP 协商结果连接起来。

SetLocalContent / SetRemoteContent

创建 channel 只是第一步，后面还要把 SDP 中的媒体参数真正 push 到 channel。

相关函数：

  • RtpTransceiver::SetChannelLocalContent

  • RtpTransceiver::SetChannelRemoteContent

它们都会进入 SetChannelContent，然后切到 worker thread 调用 channel 的 SetLocalContent 或 SetRemoteContent。

SetChannelContent 做了一个很关键的事情：它在调用 channel 的 SetLocal/SetRemoteContent 后，会重新读取 sender 的 RTP 参数并缓存回 sender。

总结

  1. RtpTransceiver 对应一个 audio/video m 行，管理 sender、receiver 和 channel。

  2. UpdateTransceiverChannel 是 SDP 应用阶段创建媒体 channel 的入口。

  3. RtpTransceiver::CreateChannel 先在 worker thread 上复用或创建 media engine 的 send/receive channel，再包装成 VoiceChannel 或 VideoChannel。

  4. RtpTransceiver::SetChannel 在 network thread 上把 VoiceChannel/VideoChannel 绑定到 RtpTransportInternal。

  5. CreateChannel 会确保 sender/receiver 持有 media channel；PushNewMediaChannel 是 SetChannel(..., set_media_channels=true) 时使用的辅助路径。

  6. SetChannelLocalContent / SetChannelRemoteContent 把 SDP 中的 codec、header extension、方向、SSRC 等媒体参数应用到 channel。
