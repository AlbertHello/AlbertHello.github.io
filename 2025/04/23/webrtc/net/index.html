webrtc 网络关键类了解

本文最后更新于 2026-06-23T17:17:19+08:00

认识 webrtc 网络关键类

  • BasicNetworkManager：全局对象。

    • 在 CreatePeerConnectionFactory() 函数中创建，具体点儿是 CreatePeerConnectionFactory() 函数创建 ConnectionContext 对象（context_）时在它的构造函数里创建的，也就是创建PeerConnectionFactory的前一步创建的。参考 peer_connection_factory的创建

    • 将来创建 BasicPortAllocator 时 BasicNetworkManager会被当做入参使用。

    • 被 ConnectionContext 持有，PeerConnectionFactory 又持有着context_

    • 在 BasicPortAllocatorSession 构造时触发 BasicNetworkManager::StartUpdating()。M146 中这里投递 UpdateNetworksContinually() task，遍历网卡并创建对应的 Network 对象。

  • BasicPacketSocketFactory：全局对象

    • 和 BasicNetworkManager 一起被创建，全局对象，被 ConnectionContext 持有。

    • 将来创建 BasicPortAllocator 时 BasicPacketSocketFactory 会被当做入参使用。BasicPortAllocator 持有它，BasicPortAllocatorSession 通过 allocator 拿到 socket_factory_，最后在 AllocationSequence::Init 中调用 CreateUdpSocket() 创建底层物理 socket。参考：webrtc物理socket的创建

  • BasicPortAllocator：

    • BasicPortAllocator 是一个子类，它继承自 PortAllocator

    • BasicPortAllocator 对象是在 PeerConnectionFactory::CreatePeerConnectionOrError 中创建的，也就是创建 peerconnection 的前一步创建的。参考peer_connection的创建

    • 同样在 peerconnection 初始化的时候 BasicPortAllocator 也初始化。

    • BasicPortAllocator 的构造函数核心入参是 Environment、BasicNetworkManager、BasicPacketSocketFactory、TurnCustomizer，其中网络管理和 socket 工厂最关键。

    • 创建 JsepTransportController 时，BasicPortAllocator 被当做入参传入，JsepTransportController 在创建P2PTransportChannel 时，再把 BasicPortAllocator 传给 p2p。

    • p2p 在 MaybeStartGathering() 函数中通过 BasicPortAllocator 调用 PortAllocator::CreateSession，函数内部再调用 BasicPortAllocator::CreateSessionInternal 函数，在该函数内创建 BasicPortAllocatorSession，BasicPortAllocatorSession 也持有了 BasicPortAllocator

  • BasicPortAllocatorSession： 网络活动相关类，通过 network thread 上的 task 串起配置、分配、候选者收集流程。

    • 它也是一个子类，继承自 PortAllocatorSession

    • 它是在 BasicPortAllocator::CreateSessionInternal() 函数创建的，是 p2p 的 MaybeStartGathering 函数通过持有的 BasicPortAllocator 触发创建的。

    • BasicPortAllocatorSession 的构造函数中通过持有的 BasicPortAllocator 拿到了 BasicNetworkManager，调用 BasicNetworkManager::StartUpdating() 开始触发创建 Network 对象。M146 中第一次更新是 PostTask(UpdateNetworksContinually)，后续每 2 秒 PostDelayedTask 重新遍历一次网卡。

    • p2p 在创建 BasicPortAllocatorSession 后，再调用 BasicPortAllocatorSession::StartGettingPorts() 函数，在该函数内 PostTask(GetPortConfigurations)，开始触发 port allocate 机制，参考 BasicPortAllocatorSession 文档中的任务机制。

    • 它在 OnAllocate() 中调用 DoAllocate()，遍历 Networks，为其创建对应的 AllocationSequence 对象，AllocationSequence 要参考 AllocationSequence 详解。

  • Network： 在 BasicNetworkManager::CreateNetworks 中被创建，其实就是一个网卡上携带的网络接口信息

    • 好理解的解释是一个网卡就是一个 Network 类，说准确点儿：因为一个网卡可能会还有多个 IP 地址也就是会有多个网络接口信息，所以最准确的就是一个 Network 类就是一个网络接口信息。通过各平台系统 API 获取网卡信息，包含ip 地址，网络类型、等很多信息。

    • 该函数会周期性调用，创建完 Network 后通过 NotifyNetworksChanged() 通知 BasicPortAllocatorSession::OnNetworksChanged()。BasicPortAllocatorSession 拿到 network 后并不是全用，会根据 allocator flags 和上层 config 过滤 network，比如 link-local、high cost、loopback、IPv6 数量限制等。

  • AllocationSequence：官方解释即按照顺序（定时）的方式为给定的网络和 IP 地址执行端口分配。

    • BasicPortAllocatorSession::DoAllocate 函数触发创建，每个 Network 对象对应一个 sequence，每个 sequence 对象又持有一个 UDPPort 对象。AllocationSequence 参考 AllocationSequence 详解。

    • AllocationSequence 能创建 UDPPort、StunPort、RelayPort 和 TCPPort，而 candidate 又是在这些 port 中创建的。port 创建完成后会通过 SubscribeCandidateReadyCallback、SubscribePortComplete、SubscribePortError 等回调通知 session，所以总体来说 AllocationSequence 就是封装了这几类 Port 对象的创建过程，便于 session 调用。

    • AllocationSequence 的 Init() 很关键：共享 UDP socket 就是在这里创建的，socket 的收包回调会先进 AllocationSequence::OnReadPacket，再由 sequence 分发给对应 Port 对象的 OnReadPacket。

  • UDPPort：UDP 类型的 ICE Port，负责创建 host candidate，并在共享 UDP socket 模式下配合 STUN server 创建 srflx candidate。

    • BasicPortAllocatorSession 创建 AllocationSequence 时，依次调用 AllocationSequence 的 Init 函数、Start 函数。M146 中 Start 会在 network thread 上投递 Process(epoch) task，开始触发创建 port 对象。

    • UDPPort、StunPort、RelayPort 和 TCPPort 都有一个类方法：Create()。创建相应的 port 对象时，都是先调用该类方法创建实例，之后再调用实例的 init 函数。

    • port 会关联一个 socket 实例，这是创建 Port 时由 sequence 传递给 port 的。port 初始化时会注册 socket 的收发、可写、本地地址等回调，后续 candidate、STUN、媒体包都会从这个 socket 进入 port。

    • 创建完 port 对象后，session 会用 PortData 持有该 port，并注册 candidate ready、candidate error、port complete、port destroyed、port error 等回调，最后触发 port 开始创建 candidate。

    • 在UDPPort::PrepareAddress()函数里面通过 socket 获取到该 socket 的本地 IP 和 端口后，开始创建 candidate 对象。

  • Candidate：信令协商候选者。有收集、排序和连通性检测三大过程。

    • 在 UDPPort::OnLocalAddressReady中拿到了本地 socket 的 ip 和 port 开始触发创建 candidate

    • 最后一步把创建完本地类型的 candidate 回调给 BasicPortAllocatorSession::OnCandidateReady

  • Connection: 代指本端端口和远端端口之间的一个连接。stun 消息、RTCP 消息和媒体数据就是从这里传递的。

    • 顾名思义就是本地端口和远端端口之间的一个连接。connection 对象是在 SetRemoteDescription 时，p2p 拿到了远端的 candidate 创建的，connection 的读写状态、建联状态都会回调给 p2p。

    • 紧接着 p2p 就会对 connection 进行排序、选择、裁剪和连通性检测，具体过程参见：ICE 建联

    • connection 在收到stun binding response 后就表示ice 建联成功了，媒体数据也就开始从远端下发了。

  • P2PTransportChannel：这是一个极其重要的类，该类管理着候选者和通道建联的过程。

    • port 创建、candidate 收集、ice 建联、媒体数据等这些状态/数据的回调都会汇集到p2p

    • offer 创建完后 SetLocalDescription 时参考SetLocalDescription，是在 JsepTransportController::CreateIceTransport 函数中被创建出来。如果 rtp 和 rtcp 是复用的，就只会创建一个 p2p。

    • p2p 被创建时 BasicPortAllocator 也被作为入参传给了 p2p。p2p 持有它是为了在 offer setLocal 成功后创建 BasicPortAllocatorSession。创建完 session 后，p2p 订阅 session 的 Port ready、candidate ready、candidate removed、candidate allocation done 等回调，然后通过 session 的 StartGettingPorts 函数开始触发 port allocate 机制。session 遍历 Networks 创建 AllocationSequence，AllocationSequence::Init 创建 socket，Start/Process 创建 port，port 再开始收集 candidate。

    • port 创建完成、candidate 收集完成回调给 session，session 再回调给 p2p

    • 开始 SetRemoteDescription 后，在 answer 里拿到了 remote candidate，p2p 准备创建 connection，并且把创建的 connection 加入到 connections_ 数组。然后设置 connection 的读写、状态、提名、错误等回调给 p2p。

    • 之后便进行 connection 排序、对选中的 connection 发起 ping，也就是通过上面创建的 socket 向远端目标地址发起 stun binding request。port 对象 OnReadPacket 函数收到远端数据，便完成了 ice 建联，有了数据通道。再区分 stun 包还是媒体包逐级往上层回调。具体过程参见SetRemoteDescription 和 ICE 建联相关文档

关系说明

  • BasicNetworkManager 不是直接从零保存所有网卡对象，它继承 NetworkManagerBase，真正复用和持有 Network 对象的是基类里的 networks_map_ 和 networks_。

  • StartUpdating() 第一次启动时记录当前 Thread，创建 PendingTaskSafetyFlag，投递 UpdateNetworksContinually()，并尝试启动 NetworkMonitorInterface。M146 中网络更新由 PostTask/PostDelayedTask + NotifyNetworksChanged() 驱动。

  • CreateNetworks() 负责调用平台 API 枚举网卡：POSIX 走 getifaddrs + ConvertIfAddrs()，Windows 走 GetAdaptersAddresses()；每个网络前缀最终会被转换成一个 Network 对象。

  • MergeNetworkList() 在 NetworkManagerBase 中做增量合并：已有 Network 会复用，新 Network 分配 id，消失的网络不会马上删除，而是标记为 inactive。

  • NetworkMonitorFactory 如果存在，会创建 NetworkMonitorInterface。它可以提供更准确的网卡类型、VPN 底层类型、网络偏好，也可以在 Android 等平台支持把 socket 绑定到指定网络。

  • SocketFactory 主要在 QueryDefaultLocalAddress() 中创建 UDP socket，连接到公共 DNS 地址来反查当前默认出口 IP，用于多网卡场景下的默认本地地址判断。

源码依据

  • webrtc_m146/src/rtc_base/network.h：NetworkManager、NetworkManagerBase、BasicNetworkManager、Network 的声明。
  • webrtc_m146/src/rtc_base/network.cc：StartUpdating()、UpdateNetworksOnce()、UpdateNetworksContinually()、CreateNetworks()、MergeNetworkList() 等实现。
  • webrtc_m146/src/rtc_base/network_monitor.h 和 webrtc_m146/src/rtc_base/network_monitor_factory.h：NetworkMonitorInterface 与 NetworkMonitorFactory 的声明。
