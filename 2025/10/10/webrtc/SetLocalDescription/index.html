webrtc之SetLocalDescription详解

本文最后更新于 2026-06-23T17:29:11+08:00

信令交换：创建完 offer 之后

SetLocalDescription 调用堆栈

  • PeerConnection::SetLocalDescription

    • SdpOfferAnswerHandler::SetLocalDescription

      • operations_chain_->ChainOperation(...) // 保证 SDP 操作按顺序执行

      • SdpOfferAnswerHandler::DoSetLocalDescription

        • ValidateSessionDescription // 检查 SDP 类型、状态、BUNDLE、m 行等是否合法

      • SdpOfferAnswerHandler::ApplyLocalDescription

        • SdpOfferAnswerHandler::PushdownTransportDescription // 创建各种 transports

          • JsepTransportController::SetLocalDescription // 切到网络线程去执行

            • JsepTransportController::ApplyDescription_n // 在网络线程

              • 准备创建 JsepTransport，一个 m 行就是一个 JsepTransport，如果是 bundled，就只有一个 JsepTransport

              • JsepTransportController::MaybeCreateJsepTransport // 一开始先根据content_info.name查询是否已经创建过

                • JsepTransportController::CreateIceTransport // 创建 P2PTransportChannel

                  • 实际创建的是P2PTransportChannel，

                  • P2PTransportChannel() // 构造，如果 rtp 和 rtcp 不复用，就会创建两个。

                    • BasicRegatheringController() // 构造

                    • P2PTransportChannel会有 ice、candidate 状态回调给JsepTransportController

                    • P2PTransportChannel中的 ice、candidate 状态来源于内部的PortAllocatorSession、Connection等网络相关类

                • JsepTransportController::CreateDtlsTransport

                  • DtlsTransport() // 构造。入参是IceTransport，持有它

                    • DtlsTransport::ConnectToIceTransport

                      • IceTransport 的读写、网络等状态回调给 DtlsTransport

                  • JsepTransportController::PrepareDtlsTransport

                    • 设置回调：IceTransport 的读写、ice、candidate 状态回调给 JsepTransportController
                    • JsepTransportController 再通过 PostTask 把这些状态回调给 PeerConnection

                • JsepTransportController::CreateUnencryptedRtpTransport // 不开启 dtls 时创建。

                  • 创建 RtpTransport，入参是rtp_packet_transport和rtcp_packet_transport，其实就是DtlsTransport
                  • RtpTransport() // 构造RtpTransport
                  • RtpTransport::SetRtpPacketTransport // 持有DtlsTransport
                    • 设置回调函数：DtlsTransport读写、网络等状态回调给 RtpTransport

                • JsepTransportController::CreateDtlsSrtpTransport // 开启 dtls 时创建

                  • DtlsSrtpTransport() // 构造DtlsSrtpTransport
                  • DtlsSrtpTransport::SetDtlsTransports
                    • 接收两个参数：rtp_dtls_transport 和 rtcp_dtls_transport，其实他俩都是前面创建的 DtlsTransport，也就持有了它。
                    • DtlsSrtpTransport::SetRtcpDtlsTransport // 只有rtcp 和 rtp 不复用时才生效
                    • RtpTransport::SetRtcpPacketTransport // 只有 rtcp 和 rtp 不复用时才生效，DtlsSrtpTransport 继承自 RtpTransport
                    • DtlsSrtpTransport::SetRtpDtlsTransport
                      • DtlsSrtpTransport::SetDtlsTransport
                        • 设置回调函数：dtls transport的 dtls 状态回调给 DtlsSrtpTransport
                    • RtpTransport::SetRtpPacketTransport // DtlsSrtpTransport 继承自 RtpTransport
                      • 设置回调：DtlsTransport读写、网络等状态回调给 RtpTransport

                • JsepTransport() 创建JsepTransport

                  • 参数是前面创建的各种 transport
                  • JsepTransport 的媒体传输状态、rtcp 状态回调到 JsepTransportController

                • JsepTransportController::CreateNetTransport

                • JsepTransportController::UpdateAggregateStates_n // 更新所有状态

              • JsepTransportDescription() // 创建JsepTransportDescription

              • JsepTransport::SetLocalJsepTransportDescription

        • UnifiedPlan 情况下：SdpOfferAnswerHandler::UpdateTransceiversAndDataChannels

          • SdpOfferAnswerHandler::AssociateTransceiver

          • SdpOfferAnswerHandler::UpdateTransceiverChannel

            • 这里不是直接 new VoiceChannel / VideoChannel，而是调用 RtpTransceiver::CreateChannel(...)。

            • RtpTransceiver::CreateChannel 会切到 worker thread，优先复用构造阶段保存的 owned_send_channel_ / owned_receive_channel_；如果没有可复用对象，再通过 media engine 创建真正的媒体收发通道：

              • audio：media_engine->voice().CreateSendChannel(...) 和 CreateReceiveChannel(...)

              • video：media_engine->video().CreateSendChannel(...) 和 CreateReceiveChannel(...)

            • 然后再把上面创建出来的 VoiceMediaSendChannelInterface / VoiceMediaReceiveChannelInterface 包装成 VoiceChannel，或者把 VideoMediaSendChannelInterface / VideoMediaReceiveChannelInterface 包装成 VideoChannel。

            • 最后 RtpTransceiver::SetChannel(...) 会切到 network thread，把 VoiceChannel/VideoChannel 绑定到前面创建好的 RtpTransportInternal。

            • M146 里更准确的链路是：UpdateTransceiverChannel -> RtpTransceiver::CreateChannel -> 复用 owned media channel 或 CreateMediaContentChannels -> CreateMediaChannel<VoiceChannel/VideoChannel> -> SetChannel。实际是 media engine 层的 send/receive media channel 先准备好，再由 VoiceChannel/VideoChannel 包装并绑定到 RtpTransport。

            • RtpTransceiver 这块比较复杂，单独看： WebRTC 之 RtpTransceiver

  * **PlanB 情况：SdpOfferAnswerHandler::CreateChannels&#x20;**

    * 旧 Plan B 路径，M146 仍有兼容代码，但主线推荐 Unified Plan。

  * **SdpOfferAnswerHandler::UpdateSessionState // 更新媒体协商状态机**

    * SdpOfferAnswerHandler::ChangeSignalingState // 更新信令状态，往上层回调

    * SdpOfferAnswerHandler::PushdownMediaDescription // 更新内部一些媒体对象

  * SdpOfferAnswerHandler::UseCandidatesInRemoteDescription

  * SdpOfferAnswerHandler::UpdateLocalSenders

* observer->OnSetLocalDescriptionComplete(OK) // 上面的 ApplyLocalDescription 函数没有错误，就代表 SetLocalDescription 成功

* JsepTransportController::MaybeStartGathering **// offer set 成功后，开始触发 candidate 收集**

  * P2PTransportChannel::MaybeStartGathering

    * PortAllocator::CreateSession

      * BasicPortAllocator::CreateSessionInternal

        * BasicPortAllocatorSession()&#x20;

          * 构造 BasicPortAllocatorSession

          * 构造函数里有两个动作：

          * 1、设置回调：network_manager的网络变化、蜂窝网络的可用不可用状态都回调到 session 类

          * 2、调用 BasicNetworkManager::StartUpdating()

            * M146 中 `StartUpdating()` 投递 `UpdateNetworksContinually()`，触发创建 Network 对象。

            * 后续网络变化通过 `NotifyNetworksChanged()` 通知 `BasicPortAllocatorSession::OnNetworksChanged()`。

        * BasicPortAllocatorSession::StartGettingPorts

          * `network_thread_->PostTask(GetPortConfigurations)`

          * 配置 ready 后投递 `OnAllocate`

          * `OnAllocate -> DoAllocate` 遍历 Network 并创建 AllocationSequence

总体描述

  • SetLocalDescription 后四大逻辑

    • PushdownTransportDescription

      • 创建几个重要的transport

      • createIceTransport：ice建联，创建 p2pTransport

      • createDtlsTransport：用于加密证书握手校验

      • createDtlsSrtpTransport：用于加密rtp包，创建的是 DtlsSrtpTransport 对象，

      • CreateUnencryptedRtpTransport：不加密时创建，创建的是 RtpTransport 对象

    • UpdateTransceiversAndDataChannels

      • UpdateTransceiverChannel 会通过 RtpTransceiver::CreateChannel 创建 VoiceChannel 或 VideoChannel，并绑定 media engine 的 send/receive channel。详细看： WebRTC 之 RtpTransceiver

    • UpdateSessionState

      • 更新媒体协商的状态机

    • MaybeStartGathering：触发 candidate 收集
