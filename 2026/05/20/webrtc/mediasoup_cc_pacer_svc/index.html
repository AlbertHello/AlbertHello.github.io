深入 mediasoup 服务端下行拥塞控制：TWCC、GoogCC、Pacer 与 SVC 分层

在 WebRTC SFU 场景里，服务端既不是简单的 RTP 转发器，也不是完整的端到端编码发送端。它需要在不转码的前提下，根据每个下行接收端的网络状态，动态决定应该发送哪些 RTP 包、保留哪些 SVC/Simulcast 层、何时降码率、何时恢复高层。

mediasoup 的设计很适合作为理解 SFU 下行抗弱网的参考样本：它把下行链路的拥塞估计交给 libwebrtc 的 GoogCC 子集处理，再把估计出来的 availableBitrate 转化为 Consumer 级别的 SVC/Simulcast 分层选择。这个闭环的核心不是“看到丢包就降层”，而是 每个 WebRtcTransport 独立做 sender-side BWE，并用带宽预算驱动分层转发。

本文从服务端下行角度拆解 mediasoup 的 CC、Pacer、BWE、SVC 分层链路。

1. 总体结论

mediasoup 的下行抗弱网可以概括为四层：

  1. 反馈输入层：客户端通过 RTCP Transport-CC、REMB、RR 等反馈下行接收状态。
  2. BWE / CC 计算层：mediasoup 内置改造过的 libwebrtc 子集，主要通过 RtpTransportControllerSend 与 GoogCcNetworkController 计算目标码率。
  3. 带宽分配层：mediasoup 将 libwebrtc 输出的 target rate 保存为 Transport 级别的 availableBitrate。
  4. 分层执行层：Transport::DistributeAvailableOutgoingBitrate() 将带宽预算分配给 Consumers，再由 SVC/Simulcast Consumer 决定目标层并执行丢层。

整体链路如下：

flowchart TD
    A["客户端接收 RTP"] --> B["RTCP Transport-CC / REMB / RR"]
    B --> C["mediasoup WebRtcTransport"]
    C --> D["TransportCongestionControlClient"]
    D --> E["libwebrtc RtpTransportControllerSend"]
    E --> F["GoogCcNetworkController"]
    F --> G["TargetTransferRate"]
    G --> H["availableBitrate"]
    H --> I["DistributeAvailableOutgoingBitrate"]
    I --> J["Consumer layer allocation"]
    J --> K["SVC / Simulcast 目标层"]
    K --> L["转发基础层或增强层 RTP"]
    L --> A

这里最重要的设计是 per-Transport 隔离。每个下行 WebRtcTransport 都有自己的 TWCC sequence、sent history、feedback adapter、GoogCC controller 和 available bitrate。某个接收端弱网时，它的反馈只会影响自己的 Transport，不会直接污染其他 Transport 的估计结果。

7.2 ALR 状态

ALR 是 Application Limited Region，表示应用层没有足够数据可发，例如静态画面、静音、视频层级已经很低等。

在 ALR 中，“没有 delay 增长”并不等于“链路容量足够”，因为发送端可能根本没有把链路打满。GoogCC 会在 ALR 中更谨慎地升码率，并通过 periodic probing 探测真实容量。

10. SVC Consumer 的关键策略

SVC Consumer 内部维护几个层级状态：

SvcConsumer::IncreaseLayer() 大致逻辑：

  1. 如果上游流不可用，返回 0。
  2. 如果已经到达 preferred layers，返回 0。
  3. 根据当前可用预算计算 virtual bitrate。
  4. 从当前 provisional layer 开始，逐个测试更高 spatial/temporal layer。
  5. 查询对应 layer bitrate。
  6. 如果预算足够，升级 provisional target layer。
  7. 返回本次升级消耗的码率。

transport-cc 模式下，mediasoup 不会在 Consumer 层再按单个 Consumer loss 做一套硬阈值降层。原因是 loss、delay、acked bitrate 已经由 GoogCC 综合处理。如果分层选择器再额外叠加独立 loss 规则，容易与 GoogCC 的判断互相打架，造成误降或震荡。

13. transport-cc 与 REMB

mediasoup 下行 BWE 优先使用 transport-cc。transport-cc 能提供 transport-wide packet arrival feedback，更适合 sender-side BWE。

当不满足 transport-cc 条件，但存在 abs-send-time 和 goog-remb RTCP feedback 时，mediasoup 可以启用 REMB 作为 fallback。

两者可以粗略比较：

机制 估计位置 输入信号 特点
Transport-CC / TWCC 发送端根据接收端反馈计算 transport-wide sequence、send time、receive delta、loss 更适合 sender-side BWE，可做 delay trend、acked bitrate、probe 分析
REMB 接收端估算后告诉发送端 receiver estimated max bitrate 较老，粒度和可控性弱于 TWCC

在现代 SFU 下行抗弱网中，应优先理解和使用 transport-cc。

16. 小结

mediasoup 下行拥塞控制的核心不是某个单独算法，而是一条完整闭环：

flowchart TD
    A["RTP 下行发送"] --> B["记录 TWCC seq / sent history"]
    B --> C["客户端 RTCP Transport Feedback"]
    C --> D["TransportFeedbackAdapter"]
    D --> E["GoogCC: delay / loss / acked / probe"]
    E --> F["TargetTransferRate"]
    F --> G["Transport availableBitrate"]
    G --> H["Consumer layer allocation"]
    H --> I["SVC / Simulcast 丢层或升层"]
    I --> A

几个关键点值得反复强调：

  1. mediasoup 的 BWE 核心来自内置的 libwebrtc GoogCC 子集，源码可见并随 worker 一起构建。
  2. 下行抗弱网以 WebRtcTransport 为隔离边界，每个接收端独立维护 BWE 状态。
  3. TWCC 提供发送时间、接收时间、丢包和 acked bitrate，是 sender-side BWE 的核心输入。
  4. GoogCC 不只看丢包，而是综合 delay trend、acked bitrate、probe、loss、RTT、ALR 等信号。
  5. BWE 输出的是 transport-level available bitrate，不是直接的 SVC 层级。
  6. SVC/Simulcast 层级由 Consumer layer allocation 执行，基础层优先，增强层按预算逐步恢复。
  7. mediasoup 的 libwebrtc pacer/prober 主要用于 probing、padding 和控制器机制；普通媒体 RTP 并不等价于完整浏览器发送端的 pacer 队列模型。

这也是 mediasoup 这套架构最值得学习的地方：它把网络估计、码率预算和分层执行解耦，让每一层都能被独立观测、独立调试，并最终形成稳定的下行抗弱网闭环。
