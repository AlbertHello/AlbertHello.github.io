rtp 传输视频数据的封装格式stap-a、fu-a

本文最后更新于 2026-05-20T21:10:39+08:00

在RTP协议中传输H.264视频数据时，支持的载荷格式包括以下几种，每种格式适用于不同的场景，具体区别如下：

1. 单一NALU包（Single NAL Unit Packet）

  • NALU类型范围：1-23、24-27（保留）、28-31（保留）
  • 结构：直接封装一个完整的NAL单元，不进行分片或聚合。
  • 适用场景：当NAL单元较小（不超过网络MTU）时使用。
  • 特点：结构简单，无额外头部开销。

2. STAP-A（Single-Time Aggregation Packet Type A）

  • NALU类型：24

  • 结构：聚合多个NALU，每个单元前加2字节的长度字段。所有单元必须属于同一RTP时间戳。

  • 适用场景：合并多个小NALU（如SPS、PPS、IDR帧），减少RTP包数量。

  • 特点：无解码顺序号（DON），要求NALU按解码顺序排列。

3. STAP-B（Single-Time Aggregation Packet Type B）

  • NALU类型：25
  • 结构：类似STAP-A，但增加2字节的**DON（Decoding Order Number）**字段。
  • 适用场景：交错传输模式（如存在B帧时），允许NALU的解码顺序与传输顺序不同。
  • 特点：支持通过DON重新排列解码顺序，复杂度较高，实际应用较少。

4. FU-A（Fragmentation Unit Type A）

  • NAL单元类型：28
  • 结构：将大NALU分片传输，每个分片包含以下字段：
    • FU指示头：类型为28，表示FU-A。
    • FU头部：包含起始（S）、结束（E）标志位及分片的NALU类型。
  • 分片规则：
    • 起始分片：S=1，携带原NAL单元类型。
    • 中间分片：S=0，E=0。
    • 结束分片：E=1。
  • 适用场景：传输超过MTU的大型NAL单元（如视频帧数据）。
  • 特点：无DON，需按传输顺序重组。

5. FU-B（Fragmentation Unit Type B）

  • NALU类型：29
  • 结构：类似FU-A，但起始分片额外包含2字节的DON字段。
  • 适用场景：交错模式下分片传输，需通过DON恢复解码顺序。
  • 特点：仅起始分片携带DON，后续分片使用FU-A结构。实际应用罕见。

各格式关键区别总结

格式 类型 分片/聚合 是否含DON 适用模式 典型场景
单一NALU包 1-23 无 无 非交错/交错模式 小数据包（如SEI、PPS）
STAP-A 24 聚合 无 非交错模式 合并多个小单元（SPS+PPS+IDR）
STAP-B 25 聚合 有 交错模式 需按DON重排序的聚合
FU-A 28 分片 无 非交错模式 大数据分片（如视频帧）
FU-B 29 分片 有 交错模式 交错模式下的分片传输

实际应用中，非交错模式（STAP-A + FU-A） 更为常见，因其实现简单；交错模式（STAP-B + FU-B）由于复杂度高，通常仅在特殊需求下使用。
