webrtc之SetRemoteDescription详解

本文最后更新于 2026-06-23T17:29:23+08:00

该篇是和SetLocalDescription对应的，当本端收到了远端的 answer，或者当远端收到了本端的 offer 之后，就会进行SetRemoteDescription 的操作。

调用堆栈标注

  • PeerConnection::SetRemoteDescription

    • SdpOfferAnswerHandler::SetRemoteDescription

      • operations_chain_->ChainOperation(...) // 和 CreateOffer、SetLocalDescription 一样，先进入操作队列

      • SdpOfferAnswerHandler::DoSetRemoteDescription

      • SdpOfferAnswerHandler::ApplyRemoteDescription

        • SdpOfferAnswerHandler::PushdownTransportDescription // 入参之一：SdpType type 表示当前是 offer 还是 answer

          • JsepTransportController::SetRemoteDescription

            • JsepTransportController::ApplyDescription_n // 转到网络线程

              • 遍历 contentInfo，也就是媒体行

              • 一个 m 行就是一个JsepTransport ，创建JsepTransport，，如果是 bundled，就只有一个JsepTransport

              • 因为 SetLocalDescription 时已经根据 content_info.name 创建了JsepTransport，并且以content_info.name(也就是 mid)为 key 保存在一个 map 中

              • JsepTransportController::MaybeCreateJsepTransport // 一开始先根据content_info.name查询是否已经创建过，所以 set offer 时已经创建了，这里直接返回。

              • 下面再次遍历contentInfo，本次遍历将和transport_infos一起工作，transport_infos就是和 ICE 相关的一个信息、ice 角色、option、用户名、密码等

              • JsepTransportController::SetIceRole_n

                • P2PTransportChannel::SetIceRole

                  • Port::SetIceRole // ICEROLE_CONTROLLING 或者 ICEROLE_CONTROLLED

              • JsepTransportController::CreateJsepTransportDescription

                • JsepTransportDescription() // 构造 JsepTransportDescription

              • JsepTransport::SetRemoteJsepTransportDescription

                • JsepTransport::SetRemoteIceParameters // 把 answer 整的 ice 相关参数存下来，比如 ice 用户名和密码、是否提名、ice mode等，

                • JsepTransport::NegotiateAndSetDtlsParameters // 拿到local 和 remote 指纹，开始协商 dtls 角色

                  • JsepTransport::NegotiateDtlsRole // 协商 dtls 角色

                  • JsepTransport::SetNegotiatedDtlsParameters // 设置 dtls 相关参数，比如 fingerprint、dtls 角色

        • UnifiedPlan 情况下：SdpOfferAnswerHandler::UpdateTransceiversAndDataChannels

          • SdpOfferAnswerHandler::AssociateTransceiver

          • SdpOfferAnswerHandler::UpdateTransceiverChannel

            • 这里会根据远端描述更新 transceiver 和媒体 channel。VoiceChannel/VideoChannel 的创建和绑定逻辑详见： WebRTC 之 RtpTransceiver

        • PlanB 情况：SdpOfferAnswerHandler::CreateChannels

          • SdpOfferAnswerHandler::RemoveUnusedChannels

        • SdpOfferAnswerHandler::UpdateSessionState // 更新媒体协商状态机

          • 如果是sdp 是 answer：SdpOfferAnswerHandler::EnableSending

            • webrtc 源码中在该函数中将上面创建的各种 Channel设置为有效

          • SdpOfferAnswerHandler::ChangeSignalingState // 更新信令状态，往上层回调

          • SdpOfferAnswerHandler::PushdownMediaDescription // 更新内部一些媒体对象

        • SdpOfferAnswerHandler::UseCandidatesInRemoteDescription // 这里开始准备创建 connection了

          • PeerConnection::UseCandidate // 把 answer 里面的 candidate 变成 candidate 对象

            • JsepTransportController::AddRemoteCandidates // 通过transport_name拿到之前创建的JsepTransport对象

              • JsepTransport::AddRemoteCandidates // 遍历刚才 answer 变成的 candidate 对象数组

                • P2PTransportChannel::AddRemoteCandidate

                  • P2PTransportChannel::FinishAddingRemoteCandidate // 入参：一个 remote_candidate

                    • P2PTransportChannel::CreateConnections // 遍历 p2p 保存的ports_数组

                      • 尝试每个 port 对象与该remote_candidate建联

                      • P2PTransportChannel::CreateConnection // 入参: 遍历到的 port 和 上面传递的 remote_candidate

                        • Port::GetConnection // 检查该 remote_candidate下是否有存在的 connection

                        • UDPPort::CreateConnection

                          • new ProxyConnection // 创建 ProxyConnection，它是 connection 的子类

                            • Connection::Connection // 构造 connection，入参：当前的 port、remote_candidate

                          • Port::AddOrReplaceConnection // Port有一个 connections_，类型是map，以远端 ip 地址保存创建的 connection

                        • P2PTransportChannel::AddConnection // p2p 有一个 connections_ 数组，保存创建的 connection

                          • 把创建的 connection 加入到 connections_ 数组

                          • 给该 connection 设置一些参数：icemode、收报超时、不可写状态超时、媒体超时等

                          • 设置connection 的回调

                            • 把connection的读写回调给 p2p。OnReadPacket、OnReadyToSend

                            • connection 的状态回调给 p2p。OnConnectionStateChange、OnConnectionDestroyed

                            • connection 被提名回调给 p2p。OnNominated

                            • connection ice 媒体状态 回调给 p2p。OnIceMediaStateChange

                            • connection stun error 回调给 p2p。OnStunError

                            • connection SocketSendError 回调给 p2p。OnSocketSendError

                            • connection PortPredictionSuccess 回调给 p2p。OnPredictionSuccess

                      • P2PTransportChannel::RememberRemoteCandidate // 把上面的remote_candidate保存起来

                      • P2PTransportChannel::SortConnectionsAndUpdateState

                      • P2PTransportChannel::UpdateConnectionStates // 遍历 p2p 的 connections_ 数组

                        • Connection::UpdateState // 每个 connection 调用自己的 UpdateState

                          • 检查 connection 的可写状态

                          • Connection::set_write_state

                          • Connection::UpdateReceiving

                      • P2PTransportChannel::CompareConnections // connection 排序

                      • P2PTransportChannel::MaybeSwitchSelectedConnection // 选出最优 connection 后，尝试切换

                      • P2PTransportChannel::PruneConnections() // 裁剪 connection

                      • P2PTransportChannel::UpdateState() // 更新本次连接的一些状态值

                      • WrappingActiveIceController::MaybeStartPinging(); // 选择可 ping 的 connection，驱动 P2PTransportChannel 发送 ping

sequenceDiagram
    participant App as 应用层
    participant SOA as SdpOfferAnswerHandler
    participant JTC as JsepTransportController
    participant P2P as P2PTransportChannel
    participant ICE as WrappingActiveIceController
    participant Port as UDPPort
    participant Conn as Connection

    App->>SOA: SetRemoteDescription(answer/offer)
    SOA->>SOA: ChainOperation + DoSetRemoteDescription
    SOA->>SOA: ApplyRemoteDescription
    SOA->>JTC: SetRemoteDescription
    JTC->>JTC: SetRemoteIceParameters + DTLS role
    SOA->>SOA: UpdateTransceiversAndDataChannels
    SOA->>SOA: UpdateSessionState
    SOA->>JTC: UseCandidatesInRemoteDescription
    JTC->>P2P: AddRemoteCandidate
    P2P->>Port: CreateConnection
    Port->>Conn: new Connection
    P2P->>P2P: SortConnectionsAndUpdateState
    P2P->>ICE: OnSortAndSwitchRequest
    ICE->>ICE: MaybeStartPinging + SelectAndPingConnection
    ICE->>P2P: SendPingRequest(connection)
    P2P->>Conn: Ping()

核心代码：

注意这里源码里写得很明确：candidate 收集只会由 SetLocalDescription 启动。SetRemoteDescription 的重点是保存远端 ICE/DTLS 参数，并把远端 candidate 用起来。

  • SetRemoteDescription

    • PushdownTransportDescription

      • 保存answer 中返回一些信息，比如远端的用户名、密码、指纹等、数据方向、最重要的就是远端candidate

    • UpdateSessionState

      • 更新媒体协商的状态机，往上层回调状态

    • UseCandidatesInRemoteDescription：这里开始准备创建 connection了
