webrtc之ICE建联模式

阅读本文之后你将收获：

  1. Candidate 到底为何物？长什么样子？怎么创建的？有什么作用？

  2. Candidate 如何排序？连通性检测又是什么？

  3. 收集到的 candidate 都是通过哪些类回调到应用层的？

ICE 建联模式

ice 建联有两种类型：

  • ICE_FULL：webrtc 的默认实现，是 ice 框架的完整版

  • ICE-LITE：ice 框架的精简版。

    • 一些 rtc 的端在有公网 ip，比如rtc 流媒体服务器，他的 ip 就直接是公网 ip 了，不需要打洞那一套。它存在公网 ip，不需要 candidate 收集，只是用 host 类型的 candidate。

    • 不需要做连通性检测，当与 ice-full 类型的端通信时，只能作为被控制端，offer 一方为controlling角色，answer一方为controlled角色，只需要 ice-full 类型的端进行连通性检测，ice-lite 一方 response 即可。当双端都是 ice-lite 类型时，都不需要连通性检测。

    • 在 sdp 中指定a=ice_mode:lite。

    • webrtc 移动端使用 ice-full 模式，mediasoup、janus都是ice-lite，也就是基本服务端都 ice-lite 模式，移动端都是 ice-full。

大概三个过程：candidate 收集、candidate 排序、连通性检测

Candidate 收集

candidate 分为以下四类，优先级由上至下：

  • Host Candidate（host）：本机所有 ip 和指定端口

    • 主机候选者指的是本地局域网内的IP地址和端口，它是直接由本地设备提供的，不需要经过任何形式的网络地址转换（NAT）或中继服务器。

    • 在WebRTC中，主机候选者的优先级通常是最高的，因为直接P2P（点对点）通信通常是最高效的方式。

    • 当两台设备位于同一局域网内时，它们将优先尝试使用主机候选者建立直接连接。

  • Server Reflexive Candidate（srflx）：服务器反射候选者。在候选收集阶段生成

    • 服务器反射候选者是通过发送STUN（Session Traversal Utilities for NAT）Binding Request给STUN服务器，并由STUN服务器返回客户端NAT映射后的外网地址得到的。

    • 此类候选者的典型场景是当设备处于 NAT 后面且需要与位于公网的对等端建立连接时，srflx 是一个关键的候选者类型。

    • 例如：candidate:1 1 UDP 1685987071 203.0.113.1 3478 typ srflx raddr 192.168.1.2 rport 5678

      • 203.0.113.1 3478分别是通过 STUN 服务器获取的公网 IP 和端口。

      • raddr 和 rport 分别是本地地址（192.168.1.2）和端口（5678）

    • 该类型候选者的优先级低于主机候选者，但高于prflx、relay 候选者。

  • Peer Reflexive Candidate（prflx）：联通测试中从对端数据报文中获取到的地址。

    • 在WebRTC连接建立过程中，当 A 端向 B 端发送数据时，如果 B 端处于 NAT 之后，那么 B 端的 NAT 设备可能会对 A 端发送的数据包进行地址和端口的转换。此时，A 端在收到的 B 端回应数据包中看到的源地址和端口，就是 B 端的 prflx Candidate。这个地址和端口是 B 端 NAT 为这次通信动态分配的公共地址和端口。

    • 双端之间直接通信时，由对等端触发生成，当一个对等端收到来自另一端点的连接请求（未在候选者列表中列出）时，会动态生成 prflx 候选者。这种候选者通常不会在初始候选列表中显示，而是在连接建立过程中生成。因此，每次通信时生成的prflx Candidate都可能不同。

    • 示例：candidate:2 1 UDP 1685987071 198.51.100.2 1234 typ prflx raddr 192.168.1.2 rport 5678

      • 198.51.100.2 1234 是对等端通过通信过程动态发现的地址。

      • raddr 和 rport 是原始本地地址及端口

  • Relay Candidate（relay）：TURN（中继）

    • 中继候选者是由TURN服务器提供的IP地址和端口，用于在两台设备无法直接建立P2P连接时，通过TURN服务器中转数据。

    • 当两台设备之间的NAT类型非常严格（如对称型NAT），导致它们无法通过主机候选者或SRFLX候选者建立连接时，将使用中继候选者通过TURN服务器中转数据。

触发收集的调用堆栈

  • PeerConnection::SetLocalDescription

    • SdpOfferAnswerHandler::SetLocalDescription

      • JsepTransportController::MaybeStartGathering // offer set 成功后，开始触发 candidate 收集

        • P2PTransportChannel::MaybeStartGathering

          • 每个媒体数据通道就是 p2pTransportChannel，但是音视频数据可以复用通道，则最终就创建一个p2pTransportChannel。

          • PortAllocator::CreateSession

            • p2p 有个成员变量allocator_，它是在构造P2PTransportChannel时传入的参数赋值的。

            • 通过调用它的createSession函数创建BasicPortAllocatorSession。

            • BasicPortAllocator::CreateSessionInternal // 创建具体的 PortAllocatorSession

              • 同时设置回调：BasicPortAllocatorSession的 IceRegathering 状态回调给 BasicPortAllocator

          • P2PTransportChannel::AddAllocatorSession

            • 创建完 BasicPortAllocatorSession 后就会把该对象添加到的P2PTransportChannel的allocator_sessions_数组中

            • 并且设置各种回调：把 PortAllocatorSession 的端口 ready、端口调整、candidate 创建、ready、remove等状态回调给 p2p

          • BasicPortAllocatorSession::StartGettingPorts // 用上面创建的session 开始 gettingPorts

            • M146 中是 network_thread_->PostTask(GetPortConfigurations)。

BasicPortAllocatorSession 端口分配机制（M146）

sequenceDiagram
    participant P2P as P2PTransportChannel
    participant S as BasicPortAllocatorSession
    participant T as network_thread
    participant Seq as AllocationSequence
    participant Port as Port

    P2P->>S: StartGettingPorts()
    S->>T: PostTask(GetPortConfigurations)
    T->>S: GetPortConfigurations()
    S->>T: PostTask(OnConfigReady)
    T->>S: OnConfigReady()
    S->>T: PostTask(OnAllocate)
    T->>S: OnAllocate()
    S->>S: DoAllocate()
    S->>Seq: Init() + Start()
    Seq->>T: PostTask(Process)
    T->>Seq: Process(UDP/Relay/TCP)
    Seq->>Port: Create + PrepareAddress
    Port-->>S: OnCandidateReady
    S-->>P2P: NotifyCandidatesReady

  • StartGettingPorts

    • 设置 session 为 GATHERING，然后投递 GetPortConfigurations()。

  • GetPortConfigurations / OnConfigReady

    • 创建 PortConfiguration，收集 STUN/TURN server、username、password 等配置。

  • AllocatePorts / OnAllocate / DoAllocate

    • 遍历每个 Network，为每个需要分配端口的 network 创建 AllocationSequence。

    • AllocationSequence 在 Init 时创建 udp socket： webrtc 之物理 socket 的创建

    • 之后调用 AllocationSequence::Start，Start 内部投递 Process。

  • AllocationSequence::Process

    • 按 UDP -> Relay -> TCP 阶段创建端口，阶段之间用 PostDelayedTask 调度。

  • OnAllocationSequenceObjectsCreated

    • 负责检查 candidate 收集是否已经完成，必要时逐级往上回调 candidate 收集完成。

排序和选择 connection

M146 中连接排序和选择主要收敛到 ICE controller 相关逻辑：WrappingActiveIceController 负责主动触发排序、切换、ping 调度，内部再包装 BasicIceController 完成候选连接的比较。可以把它理解为“从所有候选连接里挑更值得使用的那条”。核心规则没有变。

flowchart TD
    A["Connection A/B"] --> B{"写状态更好吗"}
    B -->|是| W["优先可写/更稳定连接"]
    B -->|否| C{"是否正在接收"}
    C -->|是| R["优先 receiving 连接"]
    C -->|否| D{"是否 connected"}
    D -->|是| T["TCP 等连接优先 connected"]
    D -->|否| E{"是否已提名"}
    E -->|是| N["controlled 侧优先 nominated"]
    E -->|否| F{"网络 cost/priority/generation"}
    F --> G["选择综合优先级更高的连接"]

  • 比较连接状态

    • 首先，优先选择【可写】或【假定可写】的连接，而不是【不可写】的连接。

      • 如果 a 可写，b 不可写，则 a 优于 b

      • 如果都可写：如果 a 的写状态比 b 稳定，则 a 优于 b。其写状态又分为：

        • STATE_WRITABLE：表示最近刚收到 ping response

        • STATE_WRITE_UNRELIABLE：表示有几个 ping 发送失败了

        • STATE_WRITE_INIT：表示还没收到 ping response

        • STATE_WRITE_TIMEOUT：表示有大量的 ping 发送失败了

    • 其次优先选择【可接收数据/receiving】连接，而不是【非接收/non-receiving】、【优先级/higher-priority】更高的连接

      • 如果 a 能接收数据，b 不能，则 a 优于 b

    • 最后比较是否 connected。这个情况主要针对 TCP 连接来说的。

      • 如果 a 和 b 都是可写状态，如果 a 是 connected，b 不是，即便 b 可写，也认为 a 优于 b

      • 这个判断主要避免选择 TCP 还没连上的候选对。

  • 比较是否提名。（针对 ice 被控制方 ICEROLE_CONTROLLED）

    • 如果 a 被提名，b 没有，则 a 优于 b

    • 如果都被提名了，则比较两个连接的最后接收数据的时间，越靠近现在优先级越高。

  • 比较网络情况。network cost and priority

    • 网络 cost 越低越优先，比如 Wi-Fi 通常优于蜂窝网络。

    • candidate pair priority 越高越优先。

    • generation 用于区分 ICE restart 前后的候选，新的 generation 更符合当前协商。

    • 已经被标记为 pruned 的 connection 不应该继续作为首选。

尝试切换最优 connection

  • MaybeSwitchSelectedConnection

    • 这个函数的任务是判断“新排序出来的最佳连接，是否值得替换当前 selected_connection_”。

  • 判断是否需要切换。P2PTransportChannel::ShouldSwitchSelectedConnection

    1. 如果新链接还没准备好，或者新链接本就是已选择的当前连接，则不需要切换

    2. selected_connection为空表明这是第一次建联，需要切换

    3. 比较两个 connection 的候选对儿的网络情况

    4. 再次调用CompareConnections 比较当前连接和新连接

  • 开始切换。P2PTransportChannel::SwitchSelectedConnection

    • 首先标记 route_changed = true

    • 新链接被选为selected_connection_

    • 更新network_route_，重新保存一些值。比如该新连接的本地和远端地址、网络 id、协议等

    • SignalNetworkRouteChanged 发送路由已变化的通知给 DtlsTransport::OnNetworkRouteChanged 

sequenceDiagram
    participant P2P as P2PTransportChannel
    participant Ice as IceController
    participant Conn as Connection
    participant Dtls as DtlsTransport

    P2P->>Ice: Sort/Select best connection
    Ice-->>P2P: best connection
    P2P->>P2P: ShouldSwitchSelectedConnection
    alt should switch
        P2P->>Conn: set selected_connection_
        P2P->>P2P: Update network_route_
        P2P-->>Dtls: NetworkRouteChanged
    else keep current
        P2P->>P2P: 继续使用当前连接
    end

Connection 裁剪

裁剪 PruneConnections

  • 首先针对排好序的 connections_ 数组，依次获取每个连接的网卡对象作为 key，连接本身作为 value，形成一个 map 表

    • 同一个 Network 下可能会有多条 connection，比如 host、srflx、relay 都能建联。

  • 再次遍历 connections_ 数组，把属于同一个网卡对象的连接，但却不属于最佳连接的 connection 标记裁剪

    • 裁剪的目标是减少无意义的连通性检测和保活包。

  • 裁剪。

    • 被裁剪的 connection 不再参与后续最优连接选择。
flowchart LR
    A["connections_ 已排序"] --> B["按 Network 分组"]
    B --> C["每个 Network 保留最优连接"]
    C --> D["其他同网卡连接标记 pruned"]
    D --> E["减少 ping 和状态维护成本"]

连通性检测 MaybeStartPinging

sequenceDiagram
    participant P2P as P2PTransportChannel
    participant ICE as WrappingActiveIceController
    participant Conn as Connection
    participant Req as ConnectionRequest
    participant Mgr as StunRequestManager
    participant Port as UDPPort
    participant Sock as AsyncUDPSocket

    P2P->>ICE: OnSortAndSwitchRequest()
    ICE->>ICE: MaybeStartPinging()
    ICE->>ICE: SelectAndPingConnection()
    ICE->>P2P: SendPingRequest(connection)
    P2P->>P2P: PingConnection()
    P2P->>Conn: Ping()
    Conn->>Req: BuildPingRequest()
    Conn->>Mgr: Send(request)
    Mgr->>Req: SendInternal()
    Req->>Mgr: SendPacket(buffer)
    Mgr->>Conn: send_packet_ callback
    Conn->>Port: OnSendStunPacket()
    Port->>Sock: SendTo()

  • **HasPingableConnection。**先检查 connections_ 数组中的 connection 是否可 ping。

  • **SelectAndPingConnection。**调用 BasicIceController::GetConnectionToPing() 选出本轮要 ping 的 connection。

  • **PingConnection。**开始 ping

    1. 如果是控制方

      1. 判断是否支持常规提名。本端和远端都需要支持常规提名。如果支持，则获取nomination的值。

      2. 不支持，获取 UseCandidateAttr 属性，尝试进行激进提名。也就是发起携带 USE-CANDIDATE 属性的 stun binding request

    2. 如果是被控制方。被控制方不会进行任何提名。

      1. 提名直接设置为 0 ，不进行提名

      2. use_candidate_attr = false，也不进行激进提名。

    3. 开始发送：

      • 创建一个 request：ConnectionRequest。它继承自 StunRequest

      • 再创建一个SentPing结构体：携带发送id、发送时间和表明本次 ping 是否是一个提名的标志 nomination

      • StunRequestManager::Send // 开始发送 stun binding request

        • StunRequest::Send()

          • StunRequest::SendInternal()

            • IceMessage::Write

            • StunRequestManager::SendPacket()

              • send_packet_ callback

                • Connection::OnSendStunPacket

                  • UDPPort::SendTo

                    • AsyncUDPSocket::SendTo // 物理 socket 发送

            • ConnectionRequest::OnSent

              • Connection::OnConnectionRequestSent

            • StunRequest::SendDelayed // PostDelayedTask 等待超时或重发

      • 等待 stun binding response

        • SocketDispatcher::OnEvent

          • AsyncUDPSocket::OnReadEvent

          • AsyncPacketSocket::NotifyPacketReceived

            • AllocationSequence::OnReadPacket

            • UDPPort::HandleIncomingPacket

              • UDPPort::OnReadPacket

                • Connection::OnReadPacket // 在此会进行判断是 stun 包还是媒体包

                  • 如果是 stun 包，就在该函数中继续判断 stun 包类型

                    • case STUN_BINDING_RESPONSE:

                    • case STUN_BINDING_ERROR_RESPONSE:

                      • StunRequestManager::CheckResponse()

                        • ConnectionRequest::OnResponse()

                          • Connection::OnConnectionRequestResponse

                            • Connection::ReceivedPingResponse // 收到stun response，ice 建联成功，ice 状态逐级向上回调一直到最上层。收到 response 的同时，媒体数据也同时下发了。

                            • Connection::MaybeUpdateLocalCandidate

                  • 否则就是媒体包，回调到p2p

                    • P2PTransportChannel::OnReadPacket

                      • DtlsTransport::OnReadPacket

                        • RtpTransport::OnReadPacket

                          • RtpTransport::OnRtpPacketReceived RTP 包
