webrtc之offer-answer创建

本文最后更新于 2026-06-23T17:31:37+08:00

offer/answer 的创建包括三方面的内容：证书的创建、 sdp 各相关对象构建和 sdp 序列化/反序列化

证书的创建过程（开启 DTLS 加密时需要）

  • 在 PeerConnection 构造函数中创建 SdpOfferAnswerHandler，SdpOfferAnswerHandler::Initialize 内部会创建 WebRtcSessionDescriptionFactory。

  • WebRtcSessionDescriptionFactory 构造函数中会根据 DTLS 和证书入参决定证书状态：

    • dtls_enabled == false：certificate_request_state_ = CERTIFICATE_NOT_NEEDED，并且 transport_desc_factory_ 进入 insecure 模式。

    • 外部已经传入 certificate：先设置 CERTIFICATE_WAITING，然后直接调用 SetCertificate(certificate)。

    • 需要 WebRTC 内部生成 certificate：设置 CERTIFICATE_WAITING，调用 cert_generator_->GenerateCertificateAsync(...) 异步生成证书。

sequenceDiagram
    participant PC as PeerConnection
    participant SOA as SdpOfferAnswerHandler
    participant F as WebRtcSessionDescriptionFactory
    participant G as RTCCertificateGenerator

    PC->>SOA: Create(...)
    SOA->>F: new WebRtcSessionDescriptionFactory(...)
    alt DTLS disabled
        F->>F: CERTIFICATE_NOT_NEEDED
    else certificate provided
        F->>F: SetCertificate(certificate)
    else need generate
        F->>G: GenerateCertificateAsync(callback)
        G-->>F: callback(certificate)
        F->>F: SetCertificate(certificate)
    end

核心代码：

SetCertificate() 做三件事：

  • 把状态改成 CERTIFICATE_SUCCEEDED。

  • 把证书设置到 transport_desc_factory_，后面生成 TransportDescription 时会用证书生成 fingerprint。

  • 如果证书等待期间已经有人调用 CreateOffer/CreateAnswer，这些请求会先进入 create_session_description_requests_ 队列，证书 ready 后再逐个执行。

Offer 的创建过程

  • PeerConnection::CreateOffer

    • SdpOfferAnswerHandler::CreateOffer

      • operations_chain_->ChainOperation(...) 保证 Offer/Answer/SetLocal/SetRemote 这些异步操作按顺序执行。

      • SdpOfferAnswerHandler::DoCreateOffer

        • 检查 PeerConnection 是否关闭、session error、offer options 是否合法。

        • 生成 MediaSessionOptions。

        • WebRtcSessionDescriptionFactory::CreateOffer

          • 如果证书还在等待，创建请求进入队列。

          • 如果证书已经成功，或本次不需要证书，直接进入 InternalCreateOffer()。

sequenceDiagram
    participant PC as PeerConnection
    participant SOA as SdpOfferAnswerHandler
    participant Chain as OperationsChain
    participant F as WebRtcSessionDescriptionFactory
    participant M as MediaSessionDescriptionFactory
    participant Obs as CreateSessionDescriptionObserver

    PC->>SOA: CreateOffer(observer, options)
    SOA->>Chain: ChainOperation
    Chain->>SOA: DoCreateOffer
    SOA->>F: CreateOffer(observer, options, session_options)
    alt certificate waiting
        F->>F: queue request
    else certificate ready
        F->>F: InternalCreateOffer
        F->>M: CreateOfferOrError
        M-->>F: SessionDescription
        F->>F: SessionDescriptionInterface::Create(kOffer)
        F->>Obs: PostTask(OnSuccess)
    end

核心代码：

InternalCreateOffer() 才是真正创建 SDP 对象的地方：

MediaSessionDescriptionFactory::CreateOfferOrError() 内部会创建 SDP 的三大核心对象：

  • ContentInfo：每个 m 行对应一个 content，里面持有 audio/video/data 的 media description。

  • TransportInfo：每个 m 行对应的 ICE、DTLS、fingerprint、ice ufrag/pwd 等传输参数。

  • ContentGroup：比如 BUNDLE group，用来描述多个 m 行是否复用同一个传输通道。

Offer 文本的输出

InternalCreateOffer() 创建出来的是 SessionDescriptionInterface，应用层拿到它后通常调用 ToString() 输出 SDP 文本。

M146 中成功/失败通过 signaling_thread_->PostTask(...) 回调 observer。

输出文本链路：

  • CreateSessionDescriptionObserver::OnSuccess

    • SessionDescriptionInterface::ToString

      • SdpSerialize

        • 把 SessionDescription 中的 session 级字段、每个 ContentInfo、每个 TransportInfo 序列化成 SDP 文本。

  • 应用层把 SDP 文本通过信令发送给对端。

  • 本端再调用 SetLocalDescription，开始应用本地描述并触发 candidate 收集。

Answer 的创建过程

  • WebRtcSessionDescriptionFactory::InternalCreateAnswer // 正式开始创建 answer 流程

    • MediaSessionDescriptionFactory::CreateAnswerOrError

      • auto answer = std::make_unique<SessionDescription>(); // 先创建 answer 对象

        • 开始遍历 offer 的 m 行，创建 answer 中对应的 m 行对象

        • MediaSessionDescriptionFactory::AddRtpContentForAnswer // audio/video 行

          • MediaSessionDescriptionFactory::CreateTransportAnswer // 创建 TransportDescription

            • TransportDescriptionFactory::CreateAnswer // 准备创建 answer 的 TransportDescription 对象

              • TransportDescription() // 创建 answer 的 TransportDescription 对象

              • 创建或复用 ICE 用户名、密码

              • 如果 offer 中使用了 DTLS，answer 也要携带 fingerprint，并确定 TLS 握手角色

          • NegotiateRtpTransceiverDirection // 根据 offer 方向和本端期望方向确定 answer 方向

          • AudioContentDescription / VideoContentDescription // 创建媒体描述对象

          • SetCodecsInAnswer / CreateMediaContentAnswer // 添加 codec、协议类型、rtp header extension、SSRC 等

          • MediaSessionDescriptionFactory::AddTransportAnswer // 创建 TransportInfo

            • TransportInfo 持有上面创建的 transport desc

            • SessionDescription::AddTransportInfo // 然后再把 TransportInfo 加入到 answer 对象的 transport_infos_ 数组中

          • SessionDescription::AddContent // 创建 ContentInfo

            • 创建 ContentInfo，持有上面创建的 AudioContentDescription 或 VideoContentDescription

            • 然后再把 ContentInfo 加入到 answer 对象的 contents_ 数组中

        • MediaSessionDescriptionFactory::AddDataContentForAnswer // dataChannel

          • 为 SCTP data m 行创建 SctpDataContentDescription，并补充 transport 信息

        • answer->AddGroup // 根据 offer 中的 BUNDLE 信息添加 answer 的 BUNDLE group

        • UpdateTransportInfoForBundle // 更新 BUNDLE 场景下的 Transport 参数、用户名、密码、传输方向

        • set_msid_signaling // 根据 offer 中使用的 Plan B 还是 Unified Plan 确定 answer 中媒体行组织形式

        • answer 构造完成，return answer 对象 。

      • JsepSessionDescription() // 创建Jsep对象，封装 answer 对象

      • 发送 sdp 创建成功的通知，准备往上层调用

M146 中 CreateAnswer 和 CreateOffer 一样，也会先进入 SdpOfferAnswerHandler::CreateAnswer 的 operations_chain_，再调用 DoCreateAnswer。

sequenceDiagram
    participant PC as PeerConnection
    participant SOA as SdpOfferAnswerHandler
    participant Chain as OperationsChain
    participant F as WebRtcSessionDescriptionFactory
    participant M as MediaSessionDescriptionFactory

    PC->>SOA: CreateAnswer(observer, options)
    SOA->>Chain: ChainOperation
    Chain->>SOA: DoCreateAnswer
    SOA->>SOA: 检查 signaling_state
    SOA->>SOA: GetOptionsForAnswer
    SOA->>F: CreateAnswer(observer, session_options)
    F->>M: CreateAnswerOrError
    M-->>F: SessionDescription
    F->>F: SessionDescriptionInterface::Create(kAnswer)
    F-->>PC: PostTask(observer->OnSuccess)

核心点：

  • CreateAnswer 必须在正确的信令状态下调用，M146 主要要求状态是 have-remote-offer 或 have-local-pranswer。

  • MediaSessionDescriptionFactory::CreateAnswerOrError() 会根据远端 offer 的每个 m 行创建 answer 对应的 audio/video/data content。

  • answer 的 TransportDescription 会根据 offer 的安全能力、ICE 参数、BUNDLE 关系来协商自己的 ICE/DTLS 参数。

  • 和 Offer 一样，最终用 PostCreateSessionDescriptionSucceeded() 投递 observer 成功回调。

SDP 的反序列化

即远端收到文本类型 offer 或者 answer 后需要根据文本翻译成 sdp 对象，之后才能 SetRemoteDescription

  • PeerConnectionSession::ReceiveAnswerSDP // 应用层面收到远端的 answer 文本

    • webrtc::CreateSessionDescription()

      • JsepSessionDescription() 创建 Jsep，它是对 sdp 对象的封装

      • SdpDeserialize() // SDP 开始反序列化，逐行读取文本，翻译成各种对象

        • 首先创建 sdp 对象（SessionDescription）

        • ParseSessionDescription() // 解析各种会话相关字段

        • ParseMediaDescription() // 解析媒体行

          • 遍历 m 行，准备创建对应的 audioContentDesc、videoContentDesc、TransportDesc

        • JsepSessionDescription::Initialize // 保存 sdp 对象

          • 将刚才创建的 sdp 对象保存到成员变量

          • 此时的 sdp 对象的三大成员变量：ContentGroup、ContentInfo、TransportInfo 都已经有值了。

          • 可以准备后面的SetRemoteDescription了

反序列化后的对象关系如下：
