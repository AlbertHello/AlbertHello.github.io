webrtc之STUN详解

本文最后更新于 2026-06-23T17:29:33+08:00

STUN RFC 规范：RFC 3489 和 RFC 5389 ，rfc3489 已过时，rfc5389是rfc3489 的修正版，RFC 5389 中文版参考：https://rfc2cn.com/rfc5389.html，也是目前使用最广泛的。其实rfc5389也有了修正版：rfc8489。

STUN 规范

RFC 5389 的摘要：
NAT会话遍历实用程序（STUN）是一种协议，可作为其他协议处理网络地址转换器（NAT）遍历的工具。端点可以使用它来确定NAT分配给它的IP地址和端口。它还可以用于检查两个端点之间的连接，并作为保持活动的协议来维护NAT绑定。STUN与许多现有NAT一起工作，不需要它们有任何特殊行为。

STUN本身不是NAT遍历解决方案。相反，它是在NAT遍历解决方案的上下文中使用的工具。这是对本规范先前版本（RFC 3489）的重要更改，该版本将STUN作为一个完整的解决方案。

阅读本文之后你将了解到：

STUN 的数据包分为 header 和 body 两部分，比如：

header：

字段 长度 说明
Message Type 16 bit 前两位固定为 0，剩下的 bit 表示 method 和 class
Message Length 16 bit body 长度，不包括 20 字节 header
Magic Cookie 32 bit 固定值 0x2112A442
Transaction ID 96 bit 请求和响应使用同一个事务 ID

  • stun header：20个字节

    • msg type：两个字节

      • 前两位是固定的 0

      • 有两位用于分类，C0 和 C1

      • 剩下的 12 位用于消息类型

    • msg length：两个字节。不包括消息头的长度

    • cookie：四个字节

    • transaction ID：12 个字节。 客户端StunRequest类构造时创建12 字节长度的随机字符串

Msg type

  • M0M1：代表type

  • C0C1：代表分类

    • 0b00：c0=0，c1=0代表一个请求

    • 0b01：c0=1，c1=0代表一个指示

    • 0b10：c0=0，c1=1代表成功的响应

    • 0b11：c0=1，c1=1代表失败的响应

为了十六进制计算方便，C0和C1没有挨着，比如

从M0到M3四位代表一个十六进制数

从C0到M6四位代表一个十六进制数

从C1到M9四位代表一个十六进制数

从M9到最前面省略的两个0加起来四位也代表一个十六进制数。

Transaction ID & cookie

  • cookie4 字节，固定值为0x2112a442，用来判断客户端是否支持某些属性

  • Transaction ID 12 字节，标识同一个事务的请求和响应，客户端请求和服务端响应的ID是同一个，则代表是同一事务。

stun body：Attribute（0 个或多个）

  • 0x0001 MAPPED-ADDRESS 获取客户端映射地址

  • 0x0002 RESPONSE-ADDRESS 指明对于 MAPPED-ADDRESS 的响应该由哪儿发送

  • 0x0003 CHANGE-REQUEST 请求服务端使用不同的 IP 和端口发送响应

  • 0x0004 SOURCE-ADDRESS 指定服务端的 IP 地址和端口

  • 0x0005 CHANGED-ADDRESS 是 CHANGE-REQUEST 请求的响应

  • 0x0006 USERNAME 安全验证

  • 0x0007 PASSWORD 安全验证

  • 0x0008 MESSAGE-INTEGRITY 消息完整性验证

  • 0x0009 ERROR-CODE 错误码

  • 0x000a UNKNOWN-ATTRIBUTES 未知属性

  • 0x000b REFLECTED-FROM 拒绝

以上列出的属性，并不是在任何消息类型的stun中都能加，有些属性只能出现在特定的msg type中：

标记 意义
M 必须携带
O 可选
C 条件携带
N 不可用

  • M：Must

  • N：Not Available

  • O：Optional

  • C：Client/Conditional/Compatible

每个属性都是TLV编码的：type、length、value

webrtc stun消息类型：

  • 0x0001 请求消息

  • 0x0011 指示消息

  • 0x0101 成功响应消息

  • 0x0111 错误响应消息

  • 0x0200 GOOG Ping 请求消息

  • 0x0300 GOOG Ping 响应消息

  • 0x0310 GOOG Ping 错误响应消息

webrtc 支持的 stun 属性

  • 0x0001 STUN_ATTR_MAPPED_ADDRESS 获取客户端映射地址

  • 0x0006 STUN_ATTR_USERNAME 用户名，用于安全验证

  • 0x0008 STUN_ATTR_MESSAGE_INTEGRITY 信息完整性

  • 0x0009 STUN_ATTR_ERROR_CODE 错误码

  • 0x000a STUN_ATTR_UNKNOWN_ATTRIBUTES 未知属性

  • 0x0014 STUN_ATTR_REALM 域（域名）

  • 0x0015 STUN_ATTR_NONCE 仅一次

  • 0x0020 STUN_ATTR_XOR_MAPPED_ADDRESS 异或地址

  • 0x8022 STUN_ATTR_SOFTWARE 软件

  • 0x8023 STUN_ATTR_ALTERNATE_SERVER 替代服务器

  • 0x8028 STUN_ATTR_FINGERPRINT 指纹

  • 0x802F STUN_ATTR_ORIGIN 起源

  • 0xFF00 STUN_ATTR_RETRANSMIT_COUNT Ping 重传次数

webrtc 支持的 stun 错误码

  • 300 STUN_ERROR_TRY_ALTERNATE 尝试替代服务器

  • 400 STUN_ERROR_BAD_REQUEST 错误的请求

  • 401 STUN_ERROR_UNAUTHORIZED 未授权

  • 420 STUN_ERROR_UNKNOWN_ATTRIBUTE 未知属性

  • 430 STUN_ERROR_STALE_GREDENTIALS 过期证书

  • 438 STUN_ERROR_STALE_NONCE 过期的 NONCE

  • 500 STUN_ERROR_SERVER_ERROR 服务器出错了

  • 600 STUN_ERROR_GLOBAL_FAILURE 全局失败

    Port->>Sock: SendTo()

  • P2PTransportChannel::SortConnectionsAndUpdateState
