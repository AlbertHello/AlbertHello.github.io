SDP 剖析

说到WebRTC，就要提到几个服务器：

  • Signaling 信令服务器。

    • 也就是交换会话双方的媒体信息，以及会议期间的消息，媒体描述使用的是 SDP 协议，就是按照一定规则写的一个文本，这个文本就叫sdp。
    • 可以通过http发送给对方，也可以通过udp发送给对方，至于用什么方法传给对方无所谓。

  • ICE(Interactive Connectivity Establishment)服务器

    • 中文：交互式连通性建立
    • 可以分为帮助两个客户端打洞建立 P2P 连接的 STUN 服务器，
    • 还有如果连不通就直接转发的 TURN 服务器。
    • ICE 的信息叫 Candidate，可以通过 SDP 交换，或者通过Trickle
    • 可通过WebRTC Sample了解Trickle-ice

  • SFU 或 MCU 服务器，如果多个人开会，每个端都向其他参会的端直接发送数据叫 MESH，但是 MESH 明显有局限性，SFU 就是转发可以让客户端只上行一路流转给其他客户端，而 MCU 更强大，可以上下行都只有一路流。

Global Lines

  • o=- 4611731400430051336 2 IN IP4 127.0.0.1

    • 第一个数字是会话id，是会话的唯一标识符。第二个位置的数字 2 是会话版本：如果在此媒体会话期间需要新的offer/answer协商，则该数字将增加1。当需要在媒体会话中更改任何参数（如暂停、编解码器更改、添加或删除媒体轨道）时，就会发生这种情况。以下三个字段是创建SDP的机器的网络类型IN（Internet）、IP地址类型（版本4）和单播地址。这三个值与协商无关。

  • s=-

    • s行包含一个文本会话名称，如您在此处所见，该名称并不常用。

  • t=0 0

    • 它给出了开始和结束时间。当它们都像我们的情况一样设置为0时，这意味着会话不受特定时间的限制——换句话说，它在任何时候都是永久有效的。

  • a=group:BUNDLE 0 1

    • BUNDLE 分组在 SDP 中包含的几个媒体行之间建立关系，通常是音频和视频。 在 WebRTC 中，它用于在同一个 RTP 会话中多路复用多个媒体流，如 Draft-ietf-mmusic-sdp-bundle-negotiation 中所述。 在这种情况下，浏览器提供多路复用中间 0 和 1，但这也必须得到对方的支持和接受。

  • a=msid-semantic: WMS lgsCFqt9kN2fVKw5wg3NKqGdATQoltEwOdMS

    • 此行在 PeerConnection 的生命周期中为 WebRTC 媒体流 (WMS) 提供了唯一标识符。 此标识符将用于属于特定媒体流的每个 m 行的 a=msid 属性（在我们的例子中，音频和视频 m 行）。 这意味着 RTP 媒体流（由每个 RTP 数据包中的 SSRC 字段标识）属于该媒体流，并且它是该媒体流的轨道。 它是单个 RTP 媒体流与 MediaStream WebRTC 对象的显式关联。 有关这方面的更多信息，请参阅 draft-ietf-mmusic-msid

    Audio Lines

  • m=audio 58779 UDP/TLS/RTP/SAVPF 111 103 104 9 0 8 106 105 13 126

    • m意味着它是一条媒体行——它浓缩了流媒体属性的大量信息。按照这个顺序，它告诉我们：
      • audio: 音频
      • 58779: 端口
      • UDP/TLS/RTP/SAVPF: 将用于会话的传输协议，最后但并非最不重要
      • 111 103 104 0 8 106 105 13 126：支持的媒体格式描述，以便发送和接收媒体。
    • UDP/TLS/RTP/SAVPF在RFC5764中定义。简而言之，它需要使用SRTP、SRTCP和RTCP反馈数据包。
    • 使用协议 UDP/TLS/RTP/SAVPF 的媒体格式描述给出了将用于不同格式的 RTP 有效负载编号。 低于 96 的有效载荷编号由 IANA 映射到编码格式。 在我们的 SDP 中，0 映射到 G711U，8 映射到 G711A。 大于 95 的格式编号是动态的，并且有 a=rtpmap: 属性用于从 RTP 有效负载类型编号映射到媒体编码名称。 还有 a=fmtp: 指定格式参数的属性,见图片：

  • c=IN IP4 217.130.243.155

    • c是连接行。 这一行给出了您希望发送和接收实时流量的 IP。 由于 ICE 在 WebRTC 中是强制性的，因此不会使用 c 行中的 IP。

  • a=rtcp:51472 IN IP4 217.130.243.155

    • 此行明确指定将用于 RTCP 的 IP 和端口，而不是从基本媒体端口派生的。 请注意，该端口与 SRTP 相同，因为支持 RTCP 多路复用。

ICE Candidates

WebRTC 音视频通信的双方需要知道对端的通信地址才能进行通信，WebRTC 采用 ICE 方式与通信对端建立通信连接，ICE 中很关键的一个步骤就是收集候选者信息，本端需要将自己的主机候选者，NAT 映射后的候选者以及中继候选者等信息发送给对端，对端也需要发送候选者信息给到本端，双方进行媒体连通性检测，检测成功后才能进行通信。

candidate 格式为：a=candidate:{foundation} {component} {protocol} {priority} {ip} {port} typ {type} generation {generation} ufrag {username} network_id {id} network_cost {cost}

候选者也叫 candidate，它包含一个网络地址信息，有主机候选者（host candidate），服务器反射候选者（srflx candidate），中继候选者（relay candidate）等，各种候选者代表的含义如下。

主机候选者（Local Address）是本地使用的 IP 地址和端口，例如通过 ifconfig/ipconfig 查看到 WLAN 网卡的 IP 地址是 192.168.0.105，并且准备使用 51417 端口。

服务器反射候选者（Server Reflexive Address）是 NAT 映射后使用的 IP 地址和端口。

中继候选者（Relayed Address） 是 TURN 服务器开辟的 IP 地址和端口，TURN 服务器是用来应对当通信对端 NAT 穿越失败无法建立 P2P 通信时，通过 TURN 服务器转发数据包，TURN 服务器一般都部署在带公网 IP 的机器上。

  • a=candidate:1221703924 1 udp 2122260223 192.168.0.105 51417 typ host generation 0 ufrag Q8Wv network-id 1 network-cost 10
    • typ host 表示本地候选者，使用的 IP 是 192.168.0.105，端口为 51417，使用 UDP 协议，其优先级为 2122260223，generation 表示代数，初始值为 0，如果更新 candidate 则 generation 值会递增，替换老的 candidate。
    • WebRTC example 提供了一个可以收集 candidate 的工具，通过该工具我们可以收集 Local Address 和 Server Reflexive Address，如果你搭建了 TURN 服务器，将 TURN 服务地址添加到 ICE Servers 中，你还可以收集到 Relayed Address。
  • a=candidate:1467250027 1 udp 2122260223 192.168.0.196 46243 typ host generation 0
    • ICE 是 WebRTC 中为 NAT 遍历选择的协议。 您可以在此处找到有关 ICE 的非常具有指导性和全面性的解释。 ICE 足够复杂，值得单独发表一篇文章，但我将尝试以易于理解的方式解释其 SDP 行。
    • UDP 上 RTP 的主机候选 - 在这条 ICE 行中，我们的浏览器正在为其提供主机候选 - 浏览器正在计算机上侦听的接口的 IP。 浏览器可以在该 IP 上接收/发送 SRTP 和 SRTCP，以防远程对等点的某个候选者存在 IP 可见性。 例如，如果另一台计算机在同一 LAN 上，则将使用候选主机。 协议 (udp) 后面的数字 - 2122260223 - 是候选者的优先级。 请注意，主机候选者的优先级高于其他候选者，因为使用主机候选者在资源使用方面更有效。 第一行（组件 = 1）用于 RTP，第二行（组件 = 2）用于 RTCP。
  • a=candidate:1467250027 2 udp 2122260222 192.168.0.196 56280 typ host generation 0
  • a=candidate:435653019 1 tcp 1845501695 192.168.0.196 0 typ host tcptype active generation 0
  • a=candidate:435653019 2 tcp 1845501695 192.168.0.196 0 typ host tcptype active generation 0
  • a=candidate:1853887674 1 udp 1518280447 47.61.61.61 36768 typ srflx raddr 192.168.0.196 rport 36768 generation 0
  • a=candidate:1853887674 2 udp 1518280447 47.61.61.61 36768 typ srflx raddr 192.168.0.196 rport 36768 generation 0
  • a=candidate:750991856 2 udp 25108222 237.30.30.30 51472 typ relay raddr 47.61.61.61 rport 54763 generation 0
  • a=candidate:750991856 1 udp 25108223 237.30.30.30 58779 typ relay raddr 47.61.61.61 rport 54761 generation 0

ICE Parameters

  • a=ice-ufrag:Oyef7uvBlwafI3hT
  • a=ice-pwd:T0teqPLNQQOf+5W+ls+P2p16
    • 一旦交换了 ICE 候选者，验证过程就会开始，两个浏览器都尝试使用提供的候选者相互联系。 在该过程中使用 ice-ufrag 和 ice-pwd 凭据以避免接收来自会话中未涉及的端点的潜在攻击，这些端点可能会在未经授权的情况下创建媒体会话。
